<objective>
Add rate limiting to prevent search abuse and auth endpoint brute-forcing, using the existing Upstash Redis infrastructure.

Purpose: Without rate limiting, anonymous users could hammer the search API (which calls external paid APIs) and bad actors could brute-force login. Rate limits also serve as a conversion lever -- guest limits drive sign-ups.
Output: Three rate limiters protecting search (guest + auth) and auth endpoints.
</objective>

  <name>Task 1: Rate limiter instances + proxy for auth endpoints</name>
  <files>src/lib/rate-limit.ts, src/proxy.ts</files>
  <action>
1. Create `src/lib/rate-limit.ts`:
   - Import `Ratelimit` from "@upstash/ratelimit" and `redis` from "./cache/redis"
   - Create three rate limiter instances, each conditionally (only if redis is not null):
     - `searchGuestLimiter`: sliding window, 3 requests per "1 d", prefix "rl:search:guest"
     - `searchAuthLimiter`: sliding window, 50 requests per "1 h", prefix "rl:search:auth"
     - `authEndpointLimiter`: sliding window, 5 requests per "15 m", prefix "rl:auth:endpoint"
   - If redis is null, export null for each limiter
   - Export all three

2. Create `src/proxy.ts` (Next.js 16 proxy -- replaces middleware.ts):
   - Export `async function proxy(request: NextRequest)` (NOT named "middleware")
   - Import `authEndpointLimiter` from "@/lib/rate-limit"
   - For auth endpoint requests (matched via config.matcher):
     - Extract IP from `request.headers.get("x-forwarded-for") ?? "127.0.0.1"`
     - If `authEndpointLimiter` is not null, call `.limit(ip)`
     - If not success: return `NextResponse.json({ error: "Too many attempts. Try again later." }, { status: 429 })`
     - Otherwise: return `NextResponse.next()`
   - Export config with matcher: `["/api/auth/:path*"]`
   - If authEndpointLimiter is null (no Redis), allow all requests through
  </action>

<task type="auto">

1. Modify `src/actions/search.ts`:
   - Add imports: `auth` from "@/lib/auth", `headers` from "next/headers", `searchGuestLimiter` and `searchAuthLimiter` from "@/lib/rate-limit"
   - BEFORE the existing validation logic, add rate limit check:
     - Get session: `const session = await auth.api.getSession({ headers: await headers() })`
     - Choose limiter: `session ? searchAuthLimiter : searchGuestLimiter`
     - Choose identifier: `session ? session.user.id : (reqHeaders.get("x-forwarded-for") ?? "anonymous")`
       (where reqHeaders is the resolved headers object)
     - If limiter is not null, call `.limit(identifier)`
     - If not success:
       - For guests: return `{ error: "rate_limited_guest", resetAt: reset }` (UI will show sign-up CTA)
       - For auth users: return `{ error: "rate_limited_auth", resetAt: reset }` (UI will show "try again in X minutes")
   - Keep ALL existing search logic unchanged after the rate limit check
   - Make sure to await headers() only once and reuse the result

2. Update the search form / results handling to recognize rate limit errors:
   - In the component that calls searchAction (likely the search form or its parent), check for `error === "rate_limited_guest"` or `error === "rate_limited_auth"` in the response
   - For guest: show "You've used your 3 daily searches. Sign up for more." with a link to /signup
   - For auth: compute minutes remaining from `resetAt` and show "Slow down -- try again in X minutes."
   - Find where searchAction errors are handled (likely in the search form component or the page that calls the action) and add these cases
