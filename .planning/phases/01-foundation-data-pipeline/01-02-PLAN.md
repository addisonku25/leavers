    **Step 1: Create ScrapIn validation script (scripts/validate-scrapin.ts)**

    Per locked user decision: "Validate ScrapIn API FIRST before building any UI -- write a test script that calls ScrapIn with a real company/role to see what data comes back."

    Write a standalone TypeScript script (runnable with `npx tsx scripts/validate-scrapin.ts`) that:
    1. Reads SCRAPIN_API_KEY from .env.local
    2. Calls ScrapIn Person Search endpoint with company="Google" and a role keyword
    3. Logs the raw response structure
    4. Checks if results include people who LEFT the company (not just current employees)
    5. For 2-3 results, calls Person Profile endpoint to get career history
    6. Checks if career history shows where they went AFTER Google
    7. Prints a clear PASS/FAIL verdict with reasoning

    Test these specific capabilities:
    - Can we filter by company name?
    - Do results include former employees or only current?
    - Does career history include post-departure positions?
    - What's the latency per call?
    - How many results come back?

    **Step 2: Run the validation script**

    If SCRAPIN_API_KEY is set in .env.local, run the script and document results.

    **Step 3: Implement ScrapIn provider OR document fallback**

    IF ScrapIn validation PASSES:
    - Create src/lib/data/providers/scrapin.ts implementing DataProvider interface
    - Person Search to find people at company with role
    - Person Profile for each result to get career history (parallelize with Promise.allSettled, limit to 20 profiles)
    - Extract "where they went after" from career history timeline
    - Aggregate into CareerMigration[] (destinationCompany, destinationRole, count)
    - Implement healthCheck that pings the API

    IF ScrapIn validation FAILS:
    - Document what specifically failed in a comment at top of scrapin.ts
    - Still create scrapin.ts but have it throw a descriptive error

  <behavior>
    - searchAction with valid company+role creates a search record in DB and returns searchId
    - searchAction with empty company returns validation error
    - searchAction with empty role returns validation error
    - searchAction calls getCachedOrFetch with correct params and resolved provider
    - Search record status transitions: "pending" -> "complete" (success) or "pending" -> "error" (failure)
    - searchAction stores results in migrations table linked to search record
  </behavior>
  <action>
    Write tests FIRST (RED), then implement (GREEN):

    1. Create src/lib/__tests__/search-action.test.ts testing:
       - Valid input creates search record and returns searchId
       - Invalid input (empty fields) returns Zod validation errors
       - Successful fetch updates search status to "complete"
       - Failed fetch updates search status to "error"
       Mock the database, cache manager, and provider for unit testing.

    2. Create src/actions/search.ts as a "use server" module:
       - Export async function searchAction(formData: FormData)
       - Parse and validate input with searchSchema.safeParse()
       - If validation fails, return { error: parsed.error.flatten() }
       - Generate search ID with nanoid
       - Build cache key with buildCacheKey from cache-manager
       - Create search record in Turso with status "pending"
       - Call getCachedOrFetch with params and resolved provider
       - On success: update search status to "complete", store CareerMigration[] in migrations table, redirect to /results/[searchId]
       - On error: update search status to "error", return { error: "Failed to fetch data. Please try again." }

    3. The search action should handle the case where results come from cache (fast path) vs live API fetch (slow path) -- both should work identically from the caller's perspective.

    Important: Use Next.js redirect() for the success path. The redirect throws a NEXT_REDIRECT error which is how Next.js handles server-side navigation -- do NOT catch this in try/catch.
