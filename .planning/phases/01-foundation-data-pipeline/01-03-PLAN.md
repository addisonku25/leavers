  <action>
    1. Create src/components/search-progress.tsx:
       Per locked decisions:
       - "Step-by-step progress indicators during live API fetch: named steps that check off as they complete"
       - Steps: "Querying career data..." -> "Matching role titles..." -> "Aggregating results..." (exact names are Claude's discretion)
       - "Include time expectation message: 'This may take 10-20 seconds on first search'"
       - "After 15 seconds, show 'Taking longer than usual...' but keep waiting up to 60s"
       - Use optimistic client-side timed progress per RESEARCH.md recommendation: show steps on a timer that approximates the actual process
       - Each step has a check icon when complete, a spinner when active, and is dimmed when pending
       - Visual design: clean step list with transition animations (Claude's discretion on animation style)

    2. Create src/app/results/[id]/loading.tsx:
       - Next.js loading convention file
       - Per locked decision: "Cached search results show a brief shimmer/skeleton (~500ms) before displaying -- prevents jarring instant transitions"
       - Use shadcn/ui Skeleton components in a layout that matches the results page structure

    3. Create src/app/results/[id]/page.tsx:
       - Server component that loads search record by ID from Turso
       - If search not found: show 404-style message
       - If search status is "pending": render SearchProgress component (client component for timer-based progress)
       - If search status is "complete" with results: render a basic results summary (destination companies + roles as a list/cards -- Phase 2 will add the full visualization, but this plan needs to show SOMETHING meaningful). Display: company name searched, role searched, number of results found, and a simple list of destination companies with role + count.
       - If search status is "complete" with 0 results:
         Per locked decision: "Empty results: helpful guidance with suggestions ('Try a broader role title', 'Try a larger company', 'Try a different spelling') + 'Try Another Search' button"
       - If search status is "error":
         Per locked decision: "API failure: honest error message ('Something went wrong fetching data. Please try again in a moment.') with retry button -- no silent retry"
       - Per locked decision: "Low result counts (2-3 matches): show results normally, let users judge"
       - Per locked decision: "Do NOT show cache age or 'results from X days ago' to users"
       - The "Try Another Search" button navigates back to / (landing page)

    4. Handle the pending -> complete transition:
       For live API fetches, the search action creates a "pending" record and then fetches data. The results page needs to handle this. Two approaches:
       - Option A (simpler): The search action completes the entire fetch before redirecting. The results page always sees "complete" or "error" status. The SearchProgress component is shown DURING the server action execution (before redirect) via useActionState pending state on the search form.
       - Option B (async): Search action creates pending record, redirects immediately, results page polls/refreshes until complete.

       Use Option A for v1: the search action handles everything, and the progress UI is shown on the search form page while the action runs. The results page only needs to handle complete/error states plus the skeleton loading.tsx for the brief moment while the server component loads data.

       Update the SearchForm component to show SearchProgress when the action is pending (useActionState or useTransition isPending).

    5. Create src/components/__tests__/search-progress.test.tsx:
       - Test that progress component renders all step names
       - Test that time expectation message is shown
       - Test that "taking longer than usual" appears after 15 seconds (use fake timers)
